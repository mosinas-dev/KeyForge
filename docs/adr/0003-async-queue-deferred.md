# 0003. Async-очередь (yii2-queue/Redis) — отложена (YAGNI)

## Статус

Принято.

## Контекст

KeyForge — батчевый конвейер: маркетолог загружает CSV/JSON, платформа прогоняет стадии §2 спеки (ingest → junk-фильтр → детект языка → интент → fuzzy-dedup → volume-фильтр → GAds-prep → RSA-генерация → export). Это не онлайн-нагрузка с тысячами параллельных запросов, а редкие ручные прогоны над ограниченными файлами из `sample_data/`.

Стек проекта объявляет `yii2-queue` (CLAUDE.md, §«Стек»), но с явной пометкой «отложено, §13 spec». В §13 спеки async-очередь стоит отдельной строкой среди отложенного развития:

- **Триггер «пора»:** импорт тормозит UI / объём > ~50k строк/файл.
- **Что уже заложено:** «логика в сервисе, не в контроллере».
- **Что достроить:** `yii2-queue` (Redis) + прогресс.

§7 спеки упоминает, что тяжёлый импорт/чистку *в перспективе* стоит гонять через `yii2-queue` (Redis), «чтобы UI не блокировался». Но это формулировка цели на будущее, а не задача текущих фаз. План сборки (`IMPLEMENTATION_PLAN.md`, Фаза 7) фиксирует прямо противоположное для MVP: «Тяжёлый импорт — синхронно (очередь отложена §13)».

Жёсткое правило №6 CLAUDE.md запрещает строить отложенное из §13 (явно перечислен `async`) без сработавшего триггера, квалифицируя преждевременную реализацию как нарушение YAGNI (§12 спеки: «всё из §13 не строим, пока не сработал триггер»). Принцип архитектуры — «порты сейчас, адаптеры потом» (CLAUDE.md, §«Принципы разработки»).

Решение нужно зафиксировать ADR'ом, потому что в стеке очередь упомянута, а в docker-compose уже присутствует Redis (CLAUDE.md, §«Структура проекта»: «postgres (+ redis за профилем)»). Без явной записи неизбежны попытки «достроить очередь заодно».

## Решение

1. **Async-очередь НЕ строим сейчас.** Импорт и чистка в MVP выполняются **синхронно** (Фаза 7 плана). Никаких воркеров, jobs, drivers и инфраструктуры `yii2-queue` в коде до срабатывания триггера.

2. **Закладываем порт, а не адаптер.** Бизнес-логика стадий §2 живёт в **сервисном слое** (`common/pipeline/` — `PipelineRunner`, `PipelineStage` и стадии; `common/services/`), а НЕ в теле контроллера. Консольный `KeyforgeController` (`import` / `prepare-gads` / `export`) и будущие admin-экраны остаются тонкой оркестрацией поверх сервисов. Это и есть «закладка» из §13 («логика в сервисе, не в контроллере»): вынести прогон в очередную job позже можно без переписывания стадий — меняется только точка вызова.

3. **Redis — за docker-compose профилем `queue`, не в дефолтном старте.** Обычный `docker compose up -d` (zero-touch старт по CLAUDE.md) поднимает только `php-fpm + nginx + postgres` и не запускает Redis. Контейнер Redis активируется лишь явным выбором профиля `queue`. Это согласуется с записью CLAUDE.md «Redis (очередь) и frontend-UI — за профилями/закомментированы (YAGNI)».

4. **Триггер разворота (когда строить).** Разворачиваем `yii2-queue` (Redis) + индикатор прогресса, как только выполнится любое из условий §13:
   - синхронный импорт начинает заметно блокировать/тормозить UI админки; или
   - типовой объём входного файла превышает ~50k строк.
   До этого момента — синхронно.

5. **Outbox / Saga / DAG не вводим вместе с очередью.** §12 спеки прямо говорит: батчевый пайплайн в них не нуждается; `PipelineRunner` остаётся **линейным**, не DAG (CLAUDE.md, §«Структура проекта»). Этот ADR про отложенную *очередь*, и он не открывает дверь смежному отложенному.

## Последствия

### Плюсы

- **Меньше движущихся частей в MVP.** Нет воркеров, очередей, состояний job, повторов — меньше кода, тестов и точек отказа. Соответствует YAGNI (§12) и жёсткому правилу №6.
- **Дешёвый разворот в будущем.** Логика уже в сервисном слое под интерфейсом `PipelineStage`; перенос прогона в job не затрагивает сами стадии и их тесты (§11). Стоимость закладки сейчас ≈ 0 (она и так диктуется SRP/DIP из §12).
- **Zero-touch старт не тяжелеет.** Дефолтный `docker compose up -d` не тянет Redis — быстрее подъём окружения, меньше ресурсов локально и в CI.
- **Чёткий, измеримый критерий разворота** (~50k строк / торможение UI) — решение принимается по факту, а не «на всякий случай».

### Минусы

- **Долгий синхронный импорт блокирует запрос.** На большом файле admin-загрузка (Фаза 7) будет ждать завершения прогона; нет прогресс-бара. Это осознанный компромисс MVP — приемлемо на текущих объёмах (`sample_data/` — небольшие файлы).
- **Нет фоновой обработки и ретраев.** Упавший на середине импорт перезапускается целиком. Смягчается идемпотентностью: `UNIQUE(project_id, import_hash)` и `import_hash` гарантируют, что ре-импорт того же файла даёт 0 новых строк (§9, §2.1) — повтор безопасен.

### Риски

- **Риск преждевременной достройки.** Очередь упомянута в стеке и Redis уже в compose — соблазн «включить заодно». Митигируется этим ADR + жёстким правилом №6.
- **Риск проспать триггер.** Объёмы могут вырасти незаметно, и UI начнёт тормозить раньше, чем кто-то это формализует. Митигация: при приближении входных файлов к порядку ~50k строк инициировать разворот по этому ADR. (Порог ~50k — ориентир из §13, не жёсткая константа.)
- **Риск дрейфа логики в контроллер.** Если стадии начнут «протекать» в `KeyforgeController` или admin-экраны, дешёвая закладка сломается и разворот подорожает. Митигация — ревью (A7, `code-review` skill): держать контроллеры тонкими.

## Рассмотренные альтернативы

- **Строить `yii2-queue` (Redis) сейчас, в MVP.** Отвергнуто: прямое нарушение §13 (триггер не сработал), жёсткого правила №6 и YAGNI (§12). Добавляет инфраструктуру и тесты под нагрузку, которой на текущих объёмах нет. План (Фаза 7) явно требует синхронный импорт.
- **Вообще не упоминать очередь и не закладывать сервисный слой (вся логика в контроллере).** Отвергнуто: нарушает SRP/DIP (§12) и принцип «порты сейчас, адаптеры потом». Ретрофит из контроллера в job потребовал бы переписывания — ровно то, чего §13 велит избегать дешёвой закладкой.
- **Убрать Redis из docker-compose полностью.** Отвергнуто: держать сервис за неактивным профилем `queue` — нулевая цена (не стартует при обычном `up -d`) и при этом фиксирует точку расширения. Удаление с последующим возвратом — лишняя работа без выгоды.
- **Заменить очередь самописным async (форк процесса / `proc_open` / cron-обработчик).** Отвергнуто: дублирует то, что даёт `yii2-queue`, без её гарантий; противоречит правилу №5 (не изобретать API) и плодит технический долг. Когда триггер сработает — берём штатный `yii2-queue`.
