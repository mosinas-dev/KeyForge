# 0001. Где живёт логика трансформации (SQL-first vs PHP-слой)

## Статус

Принято.

## Контекст

Конвейер KeyForge (§2 спеки) превращает сырые импортированные ключевые слова в готовые
Google Ads кампании через цепочку идемпотентных стадий: ingest и нормализация, вынос мусора
в минус-слова, детект языка, классификация интента, fuzzy-dedup, отсечка по объёму,
GAds-prep, генерация RSA, экспорт. Часть этих стадий — массовые операции над множеством
строк таблицы `kf_keyword`; часть — построчная или групповая логика, требующая обращения к
внешнему LLM и нетривиального ветвления.

Возникает архитектурный вопрос (§1 спеки): **где физически живёт логика трансформации** —
в PHP-слое (Yii2 ActiveRecord / сервисы) или в SQL-слое PostgreSQL (set-based операции,
`pg_trgm`, индексы, advisory-локи). От ответа зависят производительность на больших объёмах,
тестируемость, читаемость и распределение работы между стадиями.

Решение принимается в Фазе 1 (см. «Зафиксированные решения» плана) как фундамент для всех
последующих фаз конвейера.

Ниже — честный разбор обеих сторон (как в §1 спеки), затем вывод-гибрид.

### Аргумент за PHP-слой (Yii2 ActiveRecord / сервисы)

Вся чистка оформляется методами моделей и сервисов. Это даёт:

- лёгкое юнит-тестирование (в т.ч. через Testcontainers, §11 спеки) — чистые функции
  нормализации и классификации проверяются изолированно;
- переносимость между СУБД — логика не привязана к диалекту PostgreSQL;
- сложные правила (классификация интента по маркерам, нормализация перед fuzzy-dedup)
  выражаются императивно и читаемо;
- агенту проще генерировать код и покрывать его тестами — это прямо поддерживает TDD-режим
  и edge-кейсы §11.

Минус: построчная обработка тормозит на сотнях тысяч ключей; dedup и фильтрация по объёму
вырождаются в N циклов вместо одного множественного прохода.

### Аргумент за SQL-first (PostgreSQL, set-based)

Dedup, фильтр брендов и запрещённых, отсечка по объёму — это тривиальные множественные
операции:

- один `DELETE ... WHERE` / `UPDATE ... WHERE` вместо цикла;
- `pg_trgm` (`similarity`) для нечётких дублей прямо в БД;
- `UNIQUE`-индекс для идемпотентного импорта (`UNIQUE(project_id, import_hash)`, §8 спеки);
- advisory-локи под конкурентный импорт.

Масштабируется на миллионы строк и лежит в зоне сильной экспертизы по Postgres (MVCC,
advisory-локи).

Минус: логика размазана по миграциям/SQL, юнит-тесты тяжелее (нужен реальный Postgres),
а генерацию ad-копи всё равно делает приложение — в SQL её не унести.

## Решение

Принимается **гибрид**, ровно как зафиксировано в §1 спеки.

**SQL-first — для множественных (set-based) операций над `kf_keyword`:**

- fuzzy-dedup через `pg_trgm` (`similarity`), канон — строка с максимальным
  `search_volume` (§2.5);
- фильтры брендов / запрещённых / по объёму как множественные `WHERE`-операции (§2.6, §2.8);
- идемпотентность импорта через `UNIQUE(project_id, import_hash)` — ре-импорт того же файла
  даёт 0 новых строк (§2.1, §8, acceptance §9).

**PHP-слой — для оркестрации и логики, которая в SQL не выражается естественно:**

- оркестрация конвейера: `PipelineRunner` гоняет упорядоченный список `PipelineStage`
  (линейно, не DAG), стадии обмениваются через `PipelineContext`;
- классификация языка (детект по тексту ключа, §2.3) и интента (§2.4);
- LLM-генерация объявлений RSA с детерминированной валидацией длины 30/90 (§2.9).

**Правила-предикаты — данные в БД, не хардкод.** Бренды, запрещённые термины, пороги объёма
и карта `language → url` хранятся в таблицах `kf_config_*` (`config_brand_term`,
`config_forbidden_term`, `config_volume_threshold`, `config_language_url_map`, §8) и
сидятся миграцией. Это прямое следствие правила «правила — данные, не хардкод» (CLAUDE.md,
«Логика чистки») и закладка под отложенный UI-редактор правил (§13). SQL-фильтры читают
предикаты из этих таблиц, а не из констант в коде.

**Граница между слоями проводится по природе операции, а не по удобству:** если шаг — это
преобразование множества строк по предикату, он идёт в SQL; если это решение по одному
ключу/группе с ветвлением, обращением к LLM или вызовом внешней модели — он идёт в PHP-стадию.
Это согласуется с SRP из §12: одна стадия делает одно.

## Последствия

### Плюсы

- Производительность массовых операций: dedup и фильтры — один множественный проход,
  масштабируется на большие объёмы (снимается главный минус PHP-подхода).
- Идемпотентность импорта гарантируется схемой (`UNIQUE`), а не прикладным кодом — её нельзя
  случайно обойти (acceptance §9).
- Сложная и плохо-SQL-изируемая логика (язык, интент, LLM) остаётся в PHP, где её удобно
  юнит-тестировать и развивать (снимается главный минус SQL-подхода).
- Правила в `kf_config_*` меняются без правки кода и миграций схемы — закладка под
  админ-редактор (§13), соответствие OCP (новый предикат = новая строка конфига).
- Чёткие швы под интерфейсы §12: `PipelineStage`, `KeywordSourceProvider`, `AdCopyGenerator`,
  `CampaignExporter` — каждый слой тестируется и заменяется независимо.

### Минусы

- Логика трансформации физически живёт в двух местах (SQL-операции + PHP-стадии) — нужен
  явный навигационный ориентир (этот ADR), иначе агент/ревьюер тратит время на поиск «где
  именно».
- Часть инвариантов проверяется только на реальном Postgres: `pg_trgm` и advisory-локи на
  sqlite-моке не воспроизводятся — интеграционные тесты обязаны идти на Testcontainers
  (жёсткое правило 7 CLAUDE.md, §11). Это дороже, чем чистый юнит.
- Переносимость между СУБД сознательно принесена в жертву: привязка к PostgreSQL
  (`pg_trgm`, advisory-локи) — осознанный trade-off, не дефект.

### Риски

- **Размытие границы.** Со временем появится соблазн затащить классификацию/ветвление в SQL
  ради скорости (или наоборот — dedup в PHP-цикл). Митигация: правило границы выше + ревью
  по `code-review` skill на каждой стадии.
- **Дрейф предикатов от схемы.** Если предикат «переедет» из `kf_config_*` обратно в
  хардкод (быстрый фикс), потеряется редактируемость и возможность UI §13. Митигация:
  правило «правила — данные, не хардкод» как инвариант ревью.
- **Тестовый разрыв.** Логика в SQL легко остаётся без падающего теста, если соблазниться
  «очевидностью» запроса. Митигация: TDD строго (правило 1 CLAUDE.md) — SQL-операции
  покрываются интеграционными тестами на pgsql из §11 (порог similarity 0.85 на границе,
  tie-break канона, язык вне карты порогов и т.д.).

## Рассмотренные альтернативы

- **Вся логика в PHP-слое (чистый ActiveRecord / сервисы).** Отвергнута: построчная обработка
  тормозит на сотнях тысяч ключей, а dedup и отсечка по объёму вырождаются в N циклов вместо
  одного set-прохода (§1). Плюсы переносимости и простоты тестов не перевешивают
  деградацию на масштабе, который и есть рабочий сценарий (множественные источники ключей).
- **Вся логика в SQL-first (максимально в БД).** Отвергнута: классификация языка/интента и
  LLM-генерация объявлений в SQL не выражаются — генерацию ad-копи всё равно делает
  приложение (§1). Логика размазалась бы по миграциям, юнит-тесты усложнились бы, а часть
  шагов (LLM, §2.9) физически невозможна в БД.
- **Хардкод правил (бренды/forbidden/пороги/URL) в коде вместо `kf_config_*`.** Отвергнута:
  противоречит правилу «правила — данные, не хардкод» (CLAUDE.md) и ломает закладку под
  UI-редактор правил (§13); каждое изменение бренда/порога требовало бы правки кода и
  деплоя.
