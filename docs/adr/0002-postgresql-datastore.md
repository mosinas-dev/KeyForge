# 0002. PostgreSQL как СУБД (pg_trgm, advisory locks, Testcontainers)

## Статус

Принято.

## Контекст

KeyForge — батчевый конвейер чистки и группировки ключевых слов (§2 spec): импорт из
нескольких источников → нормализация → junk-фильтр → детект языка/интента →
fuzzy-dedup → отсечка по объёму → подготовка к Google Ads → генерация RSA → экспорт.
Ядро нагрузки на хранилище — это **множественные (set-based) операции** над
потенциально большими наборами строк, а не построчный CRUD.

Ключевые требования к хранилищу вытекают прямо из спеки:

- **Нечёткий дедуп (§2 шаг 5).** Канонизация строки и схлопывание похожих ключей по
  порогу `similarity > 0.85`; канон — строка с максимальным `search_volume`, остальные
  помечаются `merged_into_keyword_id`. Edge-кейсы (§11): точный дубль, разный регистр,
  перестановка слов, диакритика (`grátis`/`gratis`), значение ровно на пороге `0.85`,
  детерминированный tie-break при равном объёме, запрет схлопывания похожих по написанию
  ключей из **разных языков**.
- **Идемпотентность импорта (§2 шаг 1, §8, §9).** `import_hash = sha256(source + raw_keyword)`,
  ограничение `UNIQUE(project_id, import_hash)`. Повторный импорт того же файла обязан
  давать **0 новых строк** (acceptance §9), а два разных файла с одинаковым ключом —
  разные hash (§11 Ingest).
- **Конкурентный импорт.** Несколько одновременных запусков импорта не должны нарушать
  идемпотентность или плодить гонки при вставке/дедупе (§1 явно называет advisory-локи
  как механизм под конкурентный импорт).
- **Предсказуемость под нагрузкой.** Чтения (preview в админке) во время тяжёлого
  импорта/чистки не должны блокироваться и должны видеть согласованный снимок данных.

Зафиксированный стек (CLAUDE.md, §1 spec) уже называет **PostgreSQL**. Настоящий ADR
фиксирует **обоснование** этого выбора применительно к перечисленным требованиям и
закрепляет следствие для тестовой стратегии. Решение о том, где живёт логика
трансформации (гибрид SQL-first + PHP-слой), принято отдельно в §1 spec и здесь только
учитывается, а не пересматривается.

## Решение

Использовать **PostgreSQL** как единственную СУБД проекта. Конкретно опираемся на
следующие его возможности:

1. **Расширение `pg_trgm`** — для fuzzy-dedup через триграммную `similarity` с порогом
   `> 0.85` (§2 шаг 5). Расширение включается миграцией (`CREATE EXTENSION pg_trgm`,
   Фаза 1 плана) и позволяет выполнять нечёткое сопоставление как множественную SQL-операцию,
   а не построчный перебор в PHP.
2. **`UNIQUE`-индексы для идемпотентного импорта** — ограничение `UNIQUE(project_id, import_hash)`
   на `kf_keyword` (§8). Идемпотентность обеспечивается **на уровне БД** (отказ от вставки
   дубликата), а не доверяется прикладной проверке «сначала SELECT, потом INSERT», которая
   подвержена гонкам.
3. **Advisory-локи** — для сериализации конкурентного импорта (§1) без введения внешней
   инфраструктуры блокировок. Очередь/Redis отложены (§13 spec, YAGNI §12), поэтому
   блокировка «бесплатно» берётся средствами уже имеющейся СУБД.
4. **MVCC** — модель многоверсионной конкурентности Postgres даёт читателям согласованный
   снимок без блокировки писателей: preview-чтения в админке не встают в очередь за тяжёлым
   импортом. Это прямо работает на ценность «предсказуемо» (CLAUDE.md, §1 spec).

Все таблицы несут префикс `kf_` (`kf_keyword`, `kf_negative_keyword`, `kf_ad_group`,
`kf_responsive_search_ad`, `kf_import_batch`, `kf_config_*`) — CLAUDE.md.

**Следствие для тестов (закрепляется этим ADR).** Интеграционные тесты SQL-операций
(dedup, фильтры, идемпотентность, advisory-локи) выполняются на **реальном PostgreSQL
через Testcontainers**, а не на sqlite-моке. Причина прямая: `pg_trgm`-`similarity` и
advisory-локи в sqlite отсутствуют, и поведение порога `0.85`, tie-break, конкурентного
импорта на моке проверить невозможно (CLAUDE.md правило 7; §11 spec). По тест-пирамиде
(§11): ~70% unit (чистые функции нормализации/интента/валидатора RSA), ~25% integration
на реальном pgsql, ~5% e2e.

Миграции (включая `CREATE EXTENSION pg_trgm`) применяются только после ручного апрува на
shared/staging/prod; локально одноразовую БД авто-мигрирует `docker compose up -d`
(CLAUDE.md правило 2).

## Последствия

### Плюсы

- **Масштабируемость чистки.** Dedup, фильтры брендов/forbidden/объёма — один set-проход
  по индексам вместо N циклов в PHP (§1). Растёт на больших наборах ключей.
- **Идемпотентность гарантирована БД.** `UNIQUE(project_id, import_hash)` исключает дубли
  даже при гонке — не нужен прикладной «check-then-insert» (§9).
- **Конкурентный импорт без лишней инфраструктуры.** Advisory-локи дают сериализацию,
  не вытягивая в проект Redis/очередь раньше триггера (§13, §12 YAGNI).
- **Предсказуемые чтения.** MVCC: preview во время импорта не блокируется и видит
  согласованный снимок.
- **Честные тесты.** Реальный Postgres в тестах ловит то, что мок скрыл бы: диалект SQL,
  поведение `pg_trgm`, семантику advisory-локов (правило 7, §11).
- **`pg_config_*` как данные.** Реляционная модель естественно держит правила (бренды,
  forbidden, пороги, `language→url`) конфигом в таблицах, редактируемых в админке, без
  хардкода (CLAUDE.md «Логика чистки»).

### Минусы

- **Логика размазана между SQL и PHP** (осознанный trade-off §1): set-операции в
  SQL/миграциях юнит-тестируются тяжелее. Митигируется интеграционным слоем (~25%, §11).
- **Тесты требуют запущенного Postgres** (Docker/Testcontainers) — медленнее и тяжелее
  для CI, чем in-memory. Митигируется multi-stage образом с тест-гейтом и профилем
  `make test` на реальном Postgres (CLAUDE.md «Docker/CI»).
- **Привязка к Postgres.** `pg_trgm` и advisory-локи непереносимы на другую СУБД; миграция
  на иную БД потребовала бы переписать дедуп и блокировки. Принимаем — требования
  мульти-СУБД нет.
- **`CREATE EXTENSION` требует прав.** Роль, выполняющая миграцию, должна иметь право
  создавать расширение в целевой БД.

### Риски

- **Точность порога `0.85` по языкам.** Триграммная similarity может вести себя по-разному
  для разных языков/диакритики; `pg_trgm` чувствителен к акцентам, поэтому корректность
  кейса `grátis`/`gratis` (§11) зависит от нормализации/снятия диакритики на входе. Риск
  настройки порога и предобработки — см. открытые вопросы.
- **Утечка advisory-лока.** При неаккуратном управлении соединением/транзакцией лок может
  «зависнуть»; нужно явно определить уровень (session- vs transaction-scoped) и гарантию
  снятия. См. открытые вопросы.
- **Доступность Docker в CI.** Интеграционные тесты завязаны на поднятый контейнер
  Postgres; среда CI обязана это поддерживать.

## Рассмотренные альтернативы

- **SQLite / in-memory мок для тестов.** Отвергнуто: нет `pg_trgm` и advisory-локов, иной
  диалект SQL — ключевые edge-кейсы §11 (порог `0.85`, tie-break, конкурентный импорт)
  на моке непроверяемы. Прямо запрещено правилом 7 CLAUDE.md.
- **MySQL/MariaDB.** Отвергнуто: нет нативного эквивалента триграммной `similarity`
  уровня `pg_trgm`; механика блокировок и MVCC иные; экспертиза команды и весь стек уже
  на Postgres (§1). Смены СУБД требования нет.
- **Построчный fuzzy-dedup в PHP** (например, Levenshtein/набор в памяти). Отвергнуто по
  §1: построчная обработка тормозит на больших объёмах, dedup превращается в N циклов
  вместо одного set-прохода; set-based SQL масштабируется лучше.
- **Прикладная блокировка через Redis/файловый лок** под конкурентный импорт. Отвергнуто:
  тянет внешнюю зависимость (Redis отложен, §13) раньше триггера (YAGNI §12); advisory-лок
  уже доступен в выбранной СУБД и привязывает время жизни лока к сессии/транзакции.
- **Документная/NoSQL-СУБД.** Отвергнуто: модель данных §8 реляционная (FK, составные
  индексы, `UNIQUE`-ограничения, set-фильтры) ложится на реляционную СУБД естественно;
  потребности в schema-less нет.
