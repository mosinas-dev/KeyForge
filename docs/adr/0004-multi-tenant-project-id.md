# 0004. Multi-tenant: project_id закладываем, изоляцию откладываем

## Статус

Принято.

## Контекст

KeyForge сейчас обслуживает единственного заказчика — Site.pro. Однако §13 спеки прямо называет триггер «KeyForge гонят не только под Site.pro», после которого платформа должна стать multi-tenant. Multi-tenancy состоит из двух разнородных частей с принципиально разной стоимостью «доделать потом»:

1. **Форма схемы** — наличие дискриминатора арендатора (`project_id`) в каждой tenant-scoped таблице, его участие во всех составных индексах и в ограничениях уникальности. Это структура БД.
2. **Поведение** — изоляция запросов по `project_id`, выбор/переключение активного проекта, UI управления проектами, разграничение доступа. Это код и интерфейс.

Ключевое наблюдение из §8: добавить колонку `project_id` к уже наполненным таблицам задним числом дёшево только на первый взгляд. Чтобы tenant-scoped запросы оставались эффективными, **каждый составной индекс должен вести с `project_id`** (`(project_id, ...)`), а ограничения уникальности должны включать его (`UNIQUE(project_id, import_hash)` вместо `UNIQUE(import_hash)`). Ретрофит на живых данных означает: перестроить каждый составной индекс, переопределить уникальные ограничения, выполнить backfill значений и пройти миграцию на заполненных таблицах (риск простоя/блокировок). На пустой схеме в Фазе 1 (нулевое число строк) та же закладка бесплатна.

Это тот же паттерн, что зафиксирован для `import_hash` в §12 (YAGNI): «оставляем — дёшево заложить, дорого ретрофитить». Архитектурный принцип проекта — «порты сейчас, адаптеры потом» (§13): структурную закладку делаем рано, поведение разворачиваем по триггеру.

Решение уже зафиксировано как принятое в `CLAUDE.md` («Зафиксированные решения») и в `docs/IMPLEMENTATION_PLAN.md` («Зафиксированные решения», Фаза 1). Настоящий ADR фиксирует обоснование и границу: что именно закладываем сейчас и что сознательно откладываем.

## Решение

**Закладываем сейчас (Фаза 1, агент A2):**

- Справочник проектов `kf_project` (id, name). Seed: `project(id = 1, name = 'Site.pro')` — арендатор по умолчанию.
- Колонка `project_id INT NOT NULL DEFAULT 1` с внешним ключом (FK → `kf_project`) на **всех tenant-scoped таблицах**: `kf_keyword`, `kf_negative_keyword`, `kf_ad_group`, `kf_import_batch`, и каждой из `kf_config_*` (`kf_config_brand_term`, `kf_config_forbidden_term`, `kf_config_volume_threshold`, `kf_config_language_url_map`) — правила могут отличаться по проектам (§8).
- `NOT NULL DEFAULT 1` выбран намеренно: весь текущий одно-арендаторный код пишет строки, не зная о `project_id`, — каждая вставка автоматически попадает в проект 1. Никто не указывает `project_id` руками.
- **Все составные индексы ведут с `project_id`** — порядок колонок `(project_id, ...)`. Это и есть смысл ранней закладки: при включении изоляции tenant-scoped запросы сразу попадают в индекс, без перестройки.
- Идемпотентность импорта — `UNIQUE(project_id, import_hash)` (а не `UNIQUE(import_hash)`): один и тот же текст ключа из одного источника в двух разных проектах — законно разные строки.

**Область tenant-scope (по §8):**

- `kf_responsive_search_ad` **не несёт** собственного `project_id`: он привязан к группе через `ad_group_id` и арендатор-скопится транзитивно через `kf_ad_group.project_id`. Так зафиксировано в §8 и в списке tenant-scoped таблиц плана.
- `kf_project` — корень арендатора, сам колонку `project_id` не несёт.

**Откладываем до триггера «KeyForge гонят не только под Site.pro» (§13):**

- Изоляцию запросов по `project_id` (автоматическое ограничение выборок текущим проектом).
- Выбор и переключение активного проекта.
- UI управления проектами и любые экраны, завязанные на несколько арендаторов.
- Разграничение доступа к проектам (per-tenant RBAC).

До триггера система остаётся одно-арендаторной по поведению: всё неявно работает в проекте 1 за счёт `DEFAULT 1`. Реализовывать перечисленное заранее — нарушение YAGNI (§12) и пункта 6 жёстких правил `CLAUDE.md`.

## Последствия

### Плюсы

- Дорогой ретрофит (перестройка каждого составного индекса, переопределение уникальных ограничений, backfill, миграция на живых данных) исключён: закладка делается на нулевом числе строк и потому бесплатна.
- Все составные индексы уже ведут с `project_id` — при включении изоляции tenant-scoped запросы индекс-эффективны с первого дня, без миграции структуры.
- `UNIQUE(project_id, import_hash)` делает идемпотентность импорта корректной per-project с самого начала.
- `NOT NULL DEFAULT 1` оставляет весь текущий одно-арендаторный код незатронутым — он не обязан знать о multi-tenancy.
- FK гарантирует ссылочную целостность; seed `project(1, 'Site.pro')` гарантирует наличие арендатора по умолчанию.

### Минусы

- Схема несёт неиспользуемое измерение (колонка + её участие в индексах) всё время, пока арендатор один: немного «шума» в схеме и несколько байт на строку/запись индекса.
- Составные индексы, ведущие с колонкой-константой (всегда `1`), до появления реального второго арендатора менее селективны по первому полю. При текущих объёмах данных стоимость пренебрежима.

### Риски

- **Ложное ощущение «мы уже multi-tenant».** Изоляция запросов НЕ включена: запрос, забывший фильтровать по `project_id`, «работает» только из-за `DEFAULT 1` и молча сломается при появлении второго арендатора. Митигирующее условие: при срабатывании триггера обязателен отдельный проход по изоляции (скоупинг запросов + тесты на утечку между проектами); до тех пор одно-арендаторность — документированное допущение.
- **Транзитивный скоуп `kf_responsive_search_ad`.** Будущий слой изоляции обязан фильтровать RSA через join к `kf_ad_group`, а не по собственной колонке. Это нужно явно учесть при реализации изоляции (см. открытые вопросы).
- **Апрув миграции.** Миграция, добавляющая `kf_project` и `project_id`, — это изменение схемы; по пункту 2 жёстких правил `CLAUDE.md` на shared/staging/prod она применяется только после явного апрува человека. Локально `docker compose up -d` авто-мигрирует одноразовую БД (разрешено).

## Рассмотренные альтернативы

1. **Не закладывать `project_id`, ретрофитить при появлении multi-tenancy.**
   Отвергнуто: именно тот сценарий, которого §8 велит избежать — «ретрофит потребовал бы перестроить каждый индекс». На заполненных таблицах это перестройка всех составных индексов, переопределение уникальных ограничений, backfill и рискованная миграция (блокировки/простой). Дорого и рискованно потом против бесплатно сейчас (нулевое число строк).

2. **Полный multi-tenant сразу (колонка + изоляция запросов + переключение проекта + UI + per-tenant RBAC).**
   Отвергнуто: нарушение YAGNI (§12) и пункта 6 жёстких правил. Второго арендатора нет; логика изоляции, переключения и UI добавили бы сложность, поверхность тестирования и поверхность RBAC без единого текущего потребителя. Принцип «порты сейчас, адаптеры потом» (§13) прямо предписывает разворачивать адаптер по триггеру.

3. **`project_id` nullable и/или без FK.**
   Отвергнуто: `NULL` в уникальных ограничениях не равны между собой, что ломает семантику `UNIQUE(project_id, import_hash)`; nullable-колонка как ведущее поле индекса противоречит цели «вести каждый индекс с `project_id`». `NOT NULL DEFAULT 1` + FK даёт целостность и оставляет текущий код незатронутым.

4. **Изоляция по схеме/БД на арендатора (schema-per-tenant).**
   Отвергнуто как преждевременное и операционно тяжёлое. Модель общих таблиц с дискриминатором `project_id` проще, а §8 уже зафиксировал именно эту форму. Может быть пересмотрено при срабатывании триггера multi-tenant, но вне области данного ADR.
