# 0006. Решения по схеме Фазы 1: именование таблиц, import_hash, RBAC seed

**Статус:** Принято (2026-06-29, по итогам ревью ADR 0001–0005; утверждено человеком-архитектором).

## Контекст

Агент-ревьюер при проверке ADR 0001–0005 выявил три нерешённых вопроса, блокирующих DDL Фазы 1. Источник местами противоречив сам себе, поэтому требовалось явное решение человека до написания миграций (ретрофит схемы дорог: переименование таблиц, переопределение `UNIQUE`, backfill).

1. **Имя справочной таблицы проектов.** CLAUDE.md: «Префикс таблиц БД: `kf_`», но «Зафиксированные решения» и §8 пишут `project` без префикса («FK → project», «seed `project(id=1,'Site.pro')`»). ADR 0004 уже использовал `kf_project`.
2. **Семантика `import_hash`.** §2.1 даёт `sha256(source + raw_keyword)`, но «source» не определён (тип источника vs идентичность файла). §11 требует одновременно: повторный импорт того же файла → 0 новых строк И разные файлы с тем же ключом → разные хэши. Это влияет на `UNIQUE(project_id, import_hash)`.
3. **RBAC seed.** §0 относит RBAC к зоне обязательного ревью человеком; содержание ролей нигде не специфицировано.

## Решение

1. **Единый префикс `kf_` на ВСЕХ таблицах, включая `kf_project`.** Никаких исключений. FK на проект во всех tenant-scoped таблицах ведёт на `kf_project`. Это решение **supersedes** буквальные упоминания `project` без префикса в §8 и «Зафиксированных решениях» — они трактуются как сокращение, не как исключение из конвенции.

2. **`import_hash = sha256(source_type || '|' || file_hash || '|' || raw_keyword)`**, где:
   - `source_type` — тип источника (например `ahrefs_paid`, `google_ads`);
   - `file_hash` — sha256 **содержимого** импортируемого файла (хранится в `kf_import_batch.file_hash`);
   - `raw_keyword` — сырой ключ **до нормализации** (точный текст строки источника).
   - Разделитель `|` исключает склейку полей.
   - Закрепляется `UNIQUE(project_id, import_hash)`.
   - Свойства: повторный импорт того же файла → тот же `file_hash` → те же хэши → 0 новых строк (идемпотентность); два разных файла с одинаковым ключом → разные `file_hash` → разные хэши (§11 выполнено).

3. **RBAC seed (Yii2 RBAC `DbManager`, наполняется миграцией):**
   - Роль **`admin`**: полный доступ — права `manageUsers`, `manageConfig` плюс все права `marketer` (через `addChild`).
   - Роль **`marketer`**: права `importKeywords`, `reviewKeywords`, `previewCampaigns`, `exportCampaigns`.
   - Гранулярность намеренно минимальна, но реальна; привязка ролей к UI/доступу — Фаза 7. Дополнительные роли (например `viewer`) добавляются новой seed-миграцией по триггеру.

## Последствия

- **Плюсы:** единообразие схемы (вся БД под `kf_`); `import_hash` удовлетворяет обоим требованиям §11; стартовый RBAC отражает реальность (маркетолог Site.pro + админ) и не блокирует Фазу 1.
- **Минусы / риски:** `kf_project` расходится с буквой текста §8 и «Зафиксированных решений» — упоминания `project` там стоит обновить (не блокер). `file_hash` обязан вычисляться на этапе ingest и попадать в `kf_import_batch` до вычисления `import_hash` ключей — порядок стадий это учитывает.

## Рассмотренные альтернативы

- **`project` без префикса** (как в тексте фикс-решений) — отвергнуто: ломает единообразие `kf_`, ради соответствия букве, а не смыслу.
- **`import_hash = sha256(source_type + raw_keyword)`** — отвергнуто: нарушает §11 (разные файлы с тем же ключом дали бы один хэш).
- **`import_hash` по имени файла вместо содержимого** — отвергнуто: перезалитый изменённый файл с тем же именем не дал бы новых строк (ложная идемпотентность).
- **Только роль `admin`** — отвергнуто как недостаточно выразительное; **admin/marketer/viewer** — избыточно на старте (viewer вводится по необходимости).
