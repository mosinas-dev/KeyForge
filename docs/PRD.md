# PRD.md — KeyForge

Product Requirements Document. Владелец: агент A1 (Architect). Источники истины: `CLAUDE.md` (корень), `docs/AGENTIC_SPEC.md`, `docs/IMPLEMENTATION_PLAN.md`. Этот документ ничего не добавляет сверх спецификации — он переносит требования в продуктовую формулировку и фиксирует область работ.

**Статус:** Принято.

---

## 1. Проблема и боль

Маркетолог Site.pro собирает ключевые слова из нескольких источников (выгрузки Google Ads, Search Console, Ahrefs и т. п.), после чего вручную чистит их от мусора, отсекает дубли и брендовые запросы, группирует по языку и интенту и собирает из них рекламные кампании Google Ads. Эта ручная чистка и группировка, а затем сборка кампаний — медленный, утомительный и непредсказуемый процесс: результат зависит от внимательности человека, легко пропустить дубль или слить бюджет на нерелевантный запрос.

Site.pro — мультиязычный конструктор сайтов, поэтому язык ключа и целевой URL — первоклассные сущности, а не «атрибут на потом». Ошибка в языковой раскладке (ключ одного языка в группе другого, неверный целевой URL) — прямой ущерб качеству кампании.

**Что закрываем:** KeyForge автоматизирует путь от сырых ключей до готовой к импорту выгрузки Google Ads Editor. Цель — сделать этот путь **быстрее, проще и предсказуемо** (детерминированный, воспроизводимый результат), сняв с маркетолога ручную чистку, группировку и сборку RSA-объявлений.

---

## 2. Пользователь и сценарий

**Пользователь:** маркетолог Site.pro. Работает через админку (`backend`, advanced template, RBAC + AdminLTE) по адресу `http://localhost:8080/admin`.

**Основной сценарий (happy path):**

1. **Upload.** Маркетолог загружает один или несколько файлов с ключами (CSV / JSON) через форму загрузки в админке.
2. **Review.** Импортированные ключи видны в админке (Gii-CRUD ревью): нормализованный текст, определённый язык, класс интента, отметки бренд/forbidden/opportunity, статус. Мусор виден как минус-слова, дубли — схлопнутыми в канон.
3. **Preview.** Перед экспортом маркетолог открывает экран предпросмотра собранных кампаний: STAG-группы по `(intent, language)`, целевые URL, сгенерированные RSA-объявления (заголовки/описания), список минус-слов.
4. **Export.** По кнопке экспорта формируется и скачивается файл в формате Google Ads Editor, открываемый в редакторе без ошибок импорта; минус-слова выгружаются отдельно.

Тяжёлый импорт/чистка на данном этапе выполняются синхронно (асинхронная очередь отложена, см. §7). «Тестовый URL» для приёмки — локальный `http://localhost:8080/admin` после `docker compose up -d` и прогона конвейера.

## 4. Функциональные требования по стадиям конвейера (§2 spec)

Конвейер — линейная последовательность идемпотентных стадий (`PipelineRunner` прогоняет упорядоченный список `PipelineStage`, это НЕ DAG). Каждая стадия логирует in/out для предсказуемости. Команды консоли: `import` (ingest → volume-filter), `prepare-gads` (used/forbidden → группы → RSA), `export`.

1. **Ingest и нормализация.** Чтение источника (CSV/JSON через `KeywordSourceProvider`) в канонную модель `keyword`: источник, сырой и нормализованный текст, объём, язык-источника, страна, URL. `import_hash = sha256(source + raw_keyword)` обеспечивает идемпотентность: повторный импорт того же файла не создаёт дублей. Учёт прогона — в `import_batch`.
2. **Junk → минус-слова (не удалять).** Мусор (`len < 3`, только цифры/спецсимволы, профанити, нерелевантные вертикали) не удаляется, а переносится в `negative_keyword` с указанием причины и языка — он экспортируется как минус-слова Google Ads. Маркетолог получает минус-слова «бесплатно».
3. **Детект языка (не доверять источнику).** Язык определяется по тексту ключа (CLD3/fastText-подход), а не берётся из источника. Язык → язык объявления → целевой URL. Слишком короткий ключ — фолбэк на `source_language`; язык вне `language→url` карты — не падать, а флагать.
4. **Интент-классификация.** Разделение коммерческого/транзакционного и информационного интента по маркерам (`buy/create/free/best/vs/price`); спорные случаи — в LLM. В Ads уходит только коммерческий интент → меньше слитого бюджета.
5. **Fuzzy-dedup.** Нормализация (lowercase, trim, схлопывание пробелов, сортировка токенов — «website builder» == «builder website») + `pg_trgm similarity > 0.85`. Канон — строка с максимальным объёмом (детерминированный tie-break при равенстве), остальные помечаются `merged_into_keyword_id`. Похожие написания разных языков не схлопываются.
6. **Адаптивная отсечка по объёму per-language.** Порог — перцентиль внутри языка, а не глобальное значение, чтобы малые языки не вырезались несправедливо.
7. **Competitor gap.** Ключи из `ahrefs_paid` (конкуренты) МИНУС наши used — приоритетные возможности, помечаются `is_opportunity`.
8. **GAds-prep.** Убрать used (`google_ads_keywords`) и forbidden; собрать STAG-группы (single-theme ad groups) по `(intent, language)`; создать `ad_group` с `target_url` из `language→url` карты. Пустые группы не создаются; при пересечении used побеждает opportunity.
9. **RSA-генерация (LLM).** На каждую группу: 15 заголовков (≤ 30 символов) и 4 описания (≤ 90 символов) на языке группы, с подстановкой целевого URL из карты; брендовый заголовок пиннится. После генерации — детерминированная валидация длины; невалидные ответы (включая пустой/битый JSON и ответ не на языке группы) перегенерируются, а не роняют процесс. Реализуется через интерфейс `AdCopyGenerator` (`LlmAdCopyGenerator`) + `RsaLengthValidator`.
10. **Export.** Выгрузка в формате Google Ads Editor (Campaign / Ad Group / Keyword / Match Type / Final URL / Headline 1–15 / Description 1–4) через `CampaignExporter` (`GoogleAdsEditorExporter`) + preview-таблица в админке. Минус-слова экспортируются отдельно. Пустой результат даёт валидный файл с заголовком; спецсимволы экранируются; UTF-8/кириллица открывается в GAds Editor.

## 5. Нефункциональные требования

- **Предсказуемость.** Результат конвейера детерминирован и воспроизводим; каждая стадия логирует counts in/out. Конвейер собран как идемпотентные стадии.
- **Идемпотентность импорта.** Ре-импорт того же файла → 0 новых строк (двухслойная идемпотентность через `UNIQUE(project_id, import_hash)`, `import_hash = sha256(source + raw_keyword)`). Повторный `migrate` также идемпотентен.
- **Лимиты RSA.** Каждый заголовок ≤ 30 символов, каждое описание ≤ 90 символов; нарушение → перегенерация, не экспорт невалидного объявления. Лимиты проверяются детерминированным валидатором после LLM-генерации.
- **Строго моноязычные группы.** Каждая `ad_group` содержит ключи одного языка; `target_url` соответствует языку группы.
- **Правила — в `config_*`, не хардкод.** Бренды, forbidden-термины, пороги объёма и `language→url` карта хранятся в таблицах `config_brand_term`, `config_forbidden_term`, `config_volume_threshold`, `config_language_url_map` (с `project_id`), редактируются в админке. Forbidden и `language→url` сидятся заглушками (ru/en/pt/es/de → `https://site.pro/<lang>`); реальные значения заменяют seed позже и не являются блокером.
- **Тесты на реальном Postgres.** Интеграционные тесты гоняются на Testcontainers с настоящим PostgreSQL (не sqlite-мок) — иначе `pg_trgm` и advisory-локи не проверяются. Тест-пирамида: ~70% unit, ~25% integration (SQL на pgsql), ~5% e2e.
- **TDD.** Ни строки production-кода без падающего теста; порядок red → green → refactor; edge-кейсы из §11 spec пишутся первыми.
- **SQL-first для множественных операций.** Dedup (`pg_trgm`), фильтры брендов/forbidden/объёма, идемпотентность — на стороне Postgres; PHP-слой — оркестрация, классификация язык/интент, LLM-генерация.
- **Tenant-закладка.** `project_id INT NOT NULL DEFAULT 1` (FK → `project`) на всех tenant-scoped таблицах; составные индексы ведут с `project_id`; seed `project(id=1, 'Site.pro')`. Сама tenant-изоляция и UI отложены (§7).

## 6. Acceptance criteria (§9 spec)

Перенос критериев приёмки из `AGENTIC_SPEC.md` §9:

- Ре-импорт того же файла → 0 новых строк (идемпотентность).
- На вход 4 CSV из `sample_data/` → junk (`asdkjh qwe`, `????`) попадает в `negative_keyword`; бренд (`site pro builder`, `site.pro отзывы`) отфильтрован; дубли (`website builder` ×3, вариации порядка слов) схлопнуты в канон по максимальному объёму.
- Группы строго моноязычны, `target_url` соответствует языку.
- Каждый RSA-заголовок ≤ 30, описание ≤ 90 символов (иначе перегенерация).
- Экспорт открывается в Google Ads Editor без ошибок импорта.

Финальный DoD (Фаза 8): `docker compose up -d` с нуля даёт рабочую админку без ручных шагов; все ассерты §9 проходят; `make test` (полный сьют на Postgres) зелёный; CI пушит образ в `ghcr.io/<owner>/<repo>:latest`.

## 7. Вне области (отложенное, §13 spec)

Сознательно отложено по принципу YAGNI: «порт» (интерфейс) закладываем сейчас, «адаптер» строим только по триггеру. Не реализовывать заранее.

| Возможность | Триггер «пора» | Уже заложено |
|---|---|---|
| API-источники (GAds / GSC / Ahrefs) | переход с ручной выгрузки CSV на регулярный pull | интерфейс `KeywordSourceProvider` |
| Async-очередь (yii2-queue / Redis) | импорт тормозит UI / объём > ~50k строк/файл | логика в сервисе, не в контроллере |
| Multi-tenant (несколько клиентов/доменов) | KeyForge гонят не только под Site.pro | `project_id` + FK + индексы ведут с `project_id` (заложено, Фаза 1) |
| UI-редактор правил | маркетолог хочет сам менять бренды/forbidden/пороги | `config_*` как данные, не хардкод |
| Мульти-экспорт (Bing / Yandex / прямой API) | кампании не только в Google | интерфейс `CampaignExporter` |
| Смена LLM-провайдера | стоимость/качество ad-копи | интерфейс `AdCopyGenerator` |
| Дашборды наблюдаемости | нужна аналитика прогонов | структурный per-stage лог (counts in/out) |

`project_id` закладывается сейчас (колонка + FK + индексы с Фазы 1); сама tenant-изоляция запросов, выбор/переключение проекта и UI остаются отложенными до триггера. Также вне области: Outbox / Saga / DAG / multi-tenant-изоляция — батчевый пайплайн в них не нуждается.
