# KeyForge — Marketing Automation Platform (Site.pro)

**Стек:** Yii2 (advanced template) + PostgreSQL + yii2-queue
**Метод:** 0-code, всё пишут агенты через Claude Code. Человек = архитектор/ревьюер, кода руками не пишет.
**Процесс:** TDD (red → green → refactor). Тесты пишутся ПЕРВЫМИ и покрывают edge-кейсы (§11) до реализации.
**Принципы:** SOLID / DRY / YAGNI — прагматично, только где оправдано (§12). Без догматического фанатизма.
**Боль, которую закрываем:** ручная работа маркетолога по чистке/группировке ключей и сборке кампаний GAds — делаем быстрее, проще, предсказуемо.

> Допущение: Site.pro — мультиязычный конструктор сайтов, поэтому язык ключа и целевой URL — первоклассные сущности, а не атрибут «на потом». Если это не так — скажи, перестрою языковую логику.

## 1. Архитектурное решение: где живёт логика трансформации

**Аргумент за PHP-слой (Yii2 ActiveRecord).** Вся чистка как методы моделей/сервисов: легко юнит-тестировать через Testcontainers, переносимо между СУБД, сложные правила (классификация интента, fuzzy-dedup) выражаются императивно и читаемо, агенту проще генерить + покрывать тестами. Минус — построчная обработка тормозит на сотнях тысяч ключей, а dedup/фильтрация по объёму превращаются в N циклов вместо одного set-прохода.

**Аргумент за SQL-first (PostgreSQL set-based).** Dedup, фильтр брендов/запрещённых, отсечка по объёму — это тривиальные множественные операции: один `DELETE ... WHERE`, `pg_trgm` для нечётких дублей, `UNIQUE` индексы для идемпотентного импорта, advisory-локи под конкурентный импорт. Масштабируется на миллионы строк, лежит в зоне твоей экспертизы (MVCC, advisory locks). Минус — логика размазана по миграциям/SQL, юнит-тесты тяжелее, а генерацию ad-копи всё равно делает приложение.

**Вывод:** гибрид, **SQL-first для множественных операций** (dedup через `pg_trgm`, фильтры брендов/запрещённых/объёма, идемпотентность импорта через `import_hash`), **PHP-слой для оркестрации, классификации язык/интент и LLM-генерации объявлений**. Это и быстрее, и играет на твоих сильных сторонах по Postgres. Чистые правила-предикаты держим конфигом в БД (редактируются в админке), а не хардкодом.

## 2. Улучшенный алгоритм (runtime-конвейер)

Стадии как идемпотентные шаги; каждый шаг логирует in/out для предсказуемости.

1. **Ingest & нормализация** → канонная модель `keyword` (источник, сырой текст, объём, язык-источника, страна, url). `import_hash = sha256(source + raw_keyword)` → ре-импорт того же файла не плодит дублей (двухслойная идемпотентность, как в твоём Notification Service).
2. **Junk → НЕ удаляем, а в negative-список.** Мусор (`len<3`, только цифры/спецсимволы, профанити, нерелевантные вертикали) уходит в `negative_keyword`, который потом экспортируется как минус-слова GAds. Это и есть «улучшение алгоритма»: маркетолог получает минус-слова бесплатно.
3. **Детект языка (не доверять источнику).** CLD3/fastText по тексту ключа. Критично для мультиязычного Site.pro: язык → язык объявления → целевой URL.
4. **Интент-классификация.** Commercial/transactional vs informational по маркерам (`buy/create/free/best/vs/price`), спорные — в LLM. В Ads уходит только коммерческий интент → меньше слитого бюджета.
5. **Fuzzy-dedup.** Нормализация (lowercase, trim, collapse spaces, сортировка токенов: «website builder» == «builder website») + `pg_trgm similarity > 0.85`. Канон = строка с макс. объёмом, остальные → `merged_into`.
6. **Адаптивная отсечка по объёму per-language** (перцентиль внутри языка), а не глобальный порог — иначе малые языки вырезаются несправедливо.
7. **Competitor gap.** `ahrefs_paid (конкуренты)` МИНУС наши used = приоритетные ключи-возможности (помечаются `is_opportunity`).
8. **GAds-prep.** Убрать used (`google_ads_keywords`) и forbidden; сгруппировать по `(intent, language)` в STAG-группы (single-theme ad groups → выше Quality Score).
9. **RSA-генерация (LLM).** На группу: 15 заголовков (≤30 симв.) / 4 описания (≤90), на языке группы, с подстановкой целевого URL из `language→url` карты. Брендовый заголовок пиннится. После генерации — **детерминированная валидация длины**, перегенерация невалидных.
10. **Export.** CSV в формате Google Ads Editor (Campaign / Ad Group / Keyword / Match Type / Final URL / Headline 1-15 / Description 1-4) + preview-таблица в админке.

## 9. Acceptance criteria (предсказуемость)

- Ре-импорт того же файла → 0 новых строк (идемпотентность).
- На вход 4 CSV из `sample_data/` → junk (`asdkjh qwe`, `????`) в `negative_keyword`, бренд (`site pro builder`, `site.pro отзывы`) отфильтрован, дубли (`website builder` ×3, word-order вариации) схлопнуты в канон по макс. объёму.
- Группы строго моноязычны, `target_url` соответствует языку.
- Каждый RSA-заголовок ≤30, описание ≤90 символов (иначе перегенерация).
- Экспорт открывается в Google Ads Editor без ошибок импорта.

## 11. TDD: edge-кейсы по стадиям (тесты пишутся ПЕРВЫМИ)

Правило: ни строки production-кода до падающего теста. Каждая стадия §2 = свой набор. Прогон на Testcontainers (реальный pgsql, не sqlite-мок — иначе `pg_trgm`/advisory-локи не проверить). Sample-данные §sample как фикстуры happy-path; ниже — то, что они НЕ покрывают и что надо добавить руками.

| Стадия | Edge-кейсы, которые тест обязан покрыть |
|---|---|
| **Ingest/нормализация** | пустой файл; только заголовок без строк; битая кодировка (CP1251 vs UTF-8); BOM; разделитель `;` vs `,`; кавычки/запятые внутри значения; обрезанная строка (меньше колонок); лишние колонки; объём = не-число/пусто/отрицательный; повторный импорт того же файла → 0 новых (идемпотентность `import_hash`); два разных файла с одинаковым ключом → разные `import_hash` |
| **Junk → negatives** | пустая строка; `len<3`; только цифры; только спецсимволы (`????`); юникод-пробелы/zero-width; ключ из одних стоп-слов; уже помеченный negative не дублируется |
| **Детект языка** | смешанный язык в одном ключе; транслит (`sdelat sait`); слишком короткий для детекта → фолбэк на `source_language`; язык, которого нет в `language→url` карте → не падать, флагать |
| **Интент** | граничные маркеры (`free` коммерческий, `what is` информационный); ключ без маркеров → дефолт; конфликт маркеров в одной фразе |
| **Fuzzy-dedup** | точный дубль; разный регистр; перестановка слов (`website builder`/`builder builder`); диакритика (`grátis`/`gratis`); similarity ровно на пороге 0.85 (граница включения); канон = макс. объём при равных → детерминированный tie-break; разные языки с похожим написанием НЕ схлопывать |
| **Volume-фильтр** | язык с одной строкой (перцентиль не делится); все нули; язык вне карты порогов |
| **GAds-prep** | ключ есть и в used, и в opportunity (used побеждает); forbidden пересекается с opportunity; группа из одного ключа; группа, где все ключи отсеяны → не создавать пустую |
| **RSA-генерация** | заголовок ровно 30 / 31 символ (граница); описание 90/91; LLM вернул пустоту/невалидный JSON → перегенерация, не падение; язык ответа ≠ язык группы → reject |
| **Export** | спецсимволы CSV-экранируются; пустой результат → валидный файл с заголовком, не пустой байт; кириллица/UTF-8 в выгрузке открывается в GAds Editor |

Тест-пирамида: ~70% юнит на стадии (чистые функции нормализации/интента), ~25% интеграционные на SQL-операции (dedup/фильтры на pgsql), ~5% e2e — прогон 4 CSV → экспорт, ассерты из §9.

## 12. Принципы (прагматично, где оправдано)

**SOLID** — применяем точечно на швах §баланса, не размазываем по каждому классу:
- **SRP** — стадия делает одно (`JunkFilterStage` чистит, не детектит язык). Это и есть тестируемость по §11.
- **OCP** — новая стадия/источник/экспортёр = новый класс под интерфейсом, без правки существующих.
- **LSP** — любой `KeywordSourceProvider` (CSV/JSON/будущий API) взаимозаменяем в раннере.
- **ISP** — узкие интерфейсы: `KeywordSourceProvider`, `AdCopyGenerator`, `CampaignExporter` раздельны, а не один «god-интерфейс».
- **DIP** — `PipelineRunner` зависит от интерфейса `PipelineStage`, не от конкретных классов; LLM-провайдер инжектится.

Где SOLID НЕ тащим: Yii2 ActiveRecord-модели, Gii-CRUD админки, DTO/value-объекты — там абстракции были бы оверхедом.

**DRY** — нормализация ключа, валидация длины RSA, парсинг CSV — по одному месту (сервис/трейт). Но: не «дедуплицируем» случайно похожий код стадий, если у них разная причина изменения (это ложный DRY, ломает SRP).

**YAGNI** — всё из §13 не строим, пока не сработал триггер. `import_hash` оставляем (дёшево, дорого ретрофитить); Outbox/Saga/DAG НЕ закладываем — батчевый пайплайн в них не нуждается.

## 13. Будущее развитие (deferred — разворачивать по триггеру)

Отложено сознательно. Закладка-«порт» (интерфейс) есть сейчас; «адаптер» строим, когда сработает триггер. Не реализовывать заранее.

| Возможность | Триггер «пора» | Что уже заложено | Что достроить |
|---|---|---|---|
| API-источники (GAds/GSC/Ahrefs) | переход с ручной выгрузки CSV на регулярный pull | интерфейс `KeywordSourceProvider` | API-адаптеры + MCP §6 |
| Async-очередь | импорт тормозит UI / объём > ~50k строк/файл | логика в сервисе, не в контроллере | `yii2-queue` (Redis) + прогресс |
| Multi-tenant (несколько клиентов/доменов) | KeyForge гонят не только под Site.pro | колонка `project_id NOT NULL DEFAULT 1` + FK + индексы ведут с `project_id` (**заложено, Фаза 1**) | tenant-изоляция запросов + переключение/выбор проекта + UI |
| UI-редактор правил | маркетолог хочет менять бренды/forbidden/пороги сам | `config_*` как данные, не хардкод | CRUD-экран в админке |
| Мульти-экспорт (Bing/Yandex/прямой API) | кампании не только в Google | интерфейс `CampaignExporter` | новые экспортёры |
| Смена LLM-провайдера | стоимость/качество ad-копи | интерфейс `AdCopyGenerator` | новый адаптер |
| Дашборды наблюдаемости | нужна аналитика прогонов | структурный per-stage лог (counts in/out) | интеграция с AgentVault |

`project_id` — **решено: закладываем сейчас** (колонка + FK + индексы с Фазы 1). Сама tenant-изоляция и UI остаются отложенными до триггера.
